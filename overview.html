Overview¶

The application has a pipeline consisting of four stages:

  1. Discretization

  2. Code Generation

  3. Compilation

  4. Execution

BILD (Beschreibung?)

Discretization¶

Takes the input ODE system and integration method and creates a new configuration file containing sympy equations for the discretized system.

  • Input: Python module file containing the ODE system as sympy equations and the discretization method (given as a string key).

  • Output: Python module file containing the discretized ODE system as sympy equations.

  • Script for executing stage: main_disc

  • Options: discrete_records_filename, configuration_filename, methods

There are the following discretization options:

Explicit:

  • Euler method

  • Heun’s method

  • Midpoint method

  • Runge-Kutta methods (some are mathematically equivalent to methods listed above)

    • RK1 (= Euler method)

    • RK2 (= Midpoint method)

    • RK4

Implicit

  • Implicit/Backward Euler method

  • Implicit Runge-Kutta methods using a diagonalization approach (for faster integration)

    • Implicit/Backward Runge-Kutta of order 1 (mathematically equivalent to the Implicit Euler method, produces equivalent but not identical equations)

    • DIRK2 (2 stages, order 2)

    • DIRK3 (4 stages, order 3)

    • DIRK4 (3 stages, order 4)

    • Implicit Midpoint method (2 stages, order 2)

    • Gauss–Legendre method of order four (2 stages, order 4)

  • Other options

    • Strang Splitting

    • Operator Splitting

Structure / Implementation¶

The TimeIntegratorUnit class is a template for time integration Methods. This class itself cannot be instantiated. It ensures, that every method contains a term, the timestep length, a list of all variables and a method adapt_terms that returns a term consisting of only timestep specific variables (i.e. x_n) corresponding to the given term and the annotated method. This method has to be implemented in every subclass. (For Examples see any subclass of TimeIntegratorUnit.)

Different classes of integration methods are implemented as abstract subclasses of TimeIntegratorUnit that are in turn parent classes for the specific integration methods.

Extending the discretization stage / Adding new discretization methods¶

  1. Create a new subclass of TimeIntegratorUnit (or any of its subclasses).

  2. Implement the method adapt_term to that creates the discretized equations.

  3. Extend the annotation_dict in TimeIntegratorUnit with entries that connect the name of the class in this file with the keys used in the input to select the method. Example: “RungeKutta1” : RungeKutta1, “RK1”: RungeKutta1 connects the keys RungeKutta1 and RK1 to the RungeKutta1 class.

  4. Optional: Extend annotate_TIU in class ODE to create an instance of TimeIntegratorAnnotatedODE corresponding to this instance of ODE and the prior specified time integration methods. Without this the default instantiation for standard methods (e.g. the Explicit Euler method) is used.

Code Generation¶

Generate efficient code in the specified language that integrates the ODE system using the discretized equations generated in the discretization step.

  • Input: Python module file containing the ODE discretized system as sympy equations. Additional information / options (see below).

  • Options: record_filename, data_type, language, loop_type (for implicit methods), code_directory. Can be passed as arguments or given in the same file as the discretized system.

  • Output: Code in the specified language.

  • Script for executing stage: main_codegen

The code generation uses transpilers to convert the mathematical equations into code for the specified language and other options. Different languages / types of code are created by different transpilers.

Adding new transpilers¶

The Transpiler class provides an interface for all transpiler instances. Every new transpiler must be a subclass of it and implement the following methods:

  • generate(self):

  • generate_loop(self, curr_eq, loop_type, loop_iterations, condition)

  • implement_approximation_loop(self, curr_eq)

generate_loop and implement_approximation_loop are only neccessary for implicit methods. Providing an empty dummy-implementation is enough for Transpilers that are only used for explicit methods (for example Simple_C_Transpiler).

Transpilers using a code template¶

There is also an interface for transpilers that use a code template as a basis for generated code. This is given by the TemplateTranspiler class. Examples that use this are the C_Transpiler or the OMP_Transpiler.

These transpilers work by using a code template that provides a basic implementation of an ODE solver with markers at positions where code or information specific to the ODE sysstem needs to be used. These markers contain a key and are then replaced by a code module corresponding to this key. Some module - key pairs (e.g. number of equations) are already produced by the TemplateTranspiler interface while some (e.g. the code for the equations themselves) need to be added in the specific Transpilers. It is also possible to add key - value pairs by adding them to a configuration file.

Extending the code generation stage / Adding new transpilers¶

  1. Create a new subclass of Transpiler.

  2. Implement the abstract methods and properties listed in the Transpiler interface.

  3. Add code for creating and using instance of your transpiler and to the main function in main_codegen.

  4. Create a directory for the generated code (usually src/exec/<name>).

  5. Extend the compile stage with neccessary information to compile the generated code.

Compilation¶

  • Input: Code.

  • Options: …

  • Output: Executable code.

  • Script for executing stage: main_compile

Compilation is done by CompileInstance objects each having a CodeType and a Compiler attribute that contain information like file paths and flags that are needed for compilation.

To use additional compilers or code types add corresponding classes / subclasses and if neccessary extend the implementation of CompileInstance for additional functionalities.

.

Execution¶

  • Input: Executable code.

  • Options: exec_path, exec_name, input_file, dt, num_steps, output_file, begin_start_saving, num_combinations

  • Output: Solutions for the ODE system with the given parameters.

  • Script for executing stage: main_run

Extending the execution stage¶

main_run implements standard C-style execution with ./name args. Extend the main function here if other execution modes or additional arguments are needed.
